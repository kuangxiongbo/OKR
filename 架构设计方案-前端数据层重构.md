# 架构设计方案：前端数据层重构

## 一、当前方案的问题分析

### 1.1 适配层方案的问题

**方案**：应用启动时异步加载数据到 localStorage，保持同步函数不变

**问题**：
1. **数据一致性问题**
   - localStorage 和数据库可能不一致
   - 多标签页之间数据不同步
   - 数据更新后需要手动刷新

2. **实时性问题**
   - 数据变更后，其他用户/标签页看不到最新数据
   - 需要手动刷新页面才能看到更新

3. **性能问题**
   - 首次加载需要等待所有数据加载完成
   - 即使只需要部分数据，也要加载全部数据

4. **错误处理问题**
   - API 失败时，没有明确的降级策略
   - 用户可能看到过期或错误的数据

5. **可维护性问题**
   - 两套数据存储机制（localStorage + 数据库）
   - 数据同步逻辑复杂，容易出错

## 二、更优的架构方案

### 2.1 方案设计原则

1. **渐进式迁移**：最小化前端代码改动
2. **数据一致性**：单一数据源（Single Source of Truth）
3. **实时性**：数据变更后自动同步
4. **性能优化**：按需加载，智能缓存
5. **错误处理**：优雅降级，用户体验优先

### 2.3 详细设计方案

#### 方案 A：智能缓存 + 自动同步（推荐）

**核心思想**：
- 使用内存缓存作为主要数据源（快速、同步）
- localStorage 仅作为临时缓存（页面刷新时快速恢复）
- 后台自动同步，保持数据最新
- 保持原有同步 API 不变

**实现细节**：

```typescript
// services/okrService.ts

// 1. 内存缓存（主要数据源）
let usersCache: User[] = [];
let okrsCache: OKR[] = [];
let workflowsCache: ApprovalWorkflow[] = [];
// ... 其他缓存

// 2. 缓存状态管理
const cacheState = {
  users: { loading: false, lastFetch: 0, error: null },
  okrs: { loading: false, lastFetch: 0, error: null },
  // ...
};

// 3. 缓存过期时间（5分钟）
const CACHE_TTL = 5 * 60 * 1000;

// 4. 同步接口（保持原有 API 不变）
export const getUsers = (): User[] => {
  // 如果缓存为空，尝试从 localStorage 临时恢复（页面刷新场景）
  if (usersCache.length === 0) {
    const stored = localStorage.getItem(STORAGE_KEYS.USERS);
    if (stored) {
      try {
        usersCache = JSON.parse(stored);
      } catch (e) {
        console.warn('Failed to parse users from localStorage:', e);
      }
    }
  }
  
  // 后台自动刷新（如果缓存过期或为空）
  if (usersCache.length === 0 || Date.now() - cacheState.users.lastFetch > CACHE_TTL) {
    refreshUsersInBackground();
  }
  
  return usersCache.length > 0 ? usersCache : INITIAL_USERS;
};

// 5. 后台自动刷新（不阻塞 UI）
const refreshUsersInBackground = async () => {
  if (cacheState.users.loading) return;
  
  cacheState.users.loading = true;
  try {
    const result = await usersAPI.getAll();
    if (result.success && result.data?.users) {
      usersCache = result.data.users;
      localStorage.setItem(STORAGE_KEYS.USERS, JSON.stringify(usersCache));
      cacheState.users.lastFetch = Date.now();
      cacheState.users.error = null;
      dispatchUpdate(); // 通知 UI 更新
    }
  } catch (error) {
    cacheState.users.error = error;
    // API 失败时，如果缓存为空，尝试从 localStorage 恢复（仅用于页面刷新场景）
    if (usersCache.length === 0) {
      const stored = localStorage.getItem(STORAGE_KEYS.USERS);
      if (stored) {
        try {
          usersCache = JSON.parse(stored);
        } catch (e) {
          console.warn('Failed to parse users from localStorage:', e);
        }
      }
    }
    // 如果仍然为空，使用初始数据
    if (usersCache.length === 0) {
      usersCache = INITIAL_USERS;
    }
  } finally {
    cacheState.users.loading = false;
  }
};

// 6. 初始化函数（应用启动时调用）
export const initializeData = async () => {
  // 先尝试从 localStorage 恢复（页面刷新时快速显示，提升用户体验）
  try {
    const storedUsers = localStorage.getItem(STORAGE_KEYS.USERS);
    if (storedUsers) {
      usersCache = JSON.parse(storedUsers);
    }
    // ... 其他数据类似处理
  } catch (e) {
    console.warn('Failed to restore data from localStorage:', e);
  }
  
  // 然后立即从 API 刷新（获取最新数据，覆盖缓存）
  await Promise.all([
    refreshUsersInBackground(),
    refreshOKRsInBackground(),
    // ... 其他数据
  ]);
};

// 7. 保存操作（同时更新缓存和 API）
export const saveOKR = async (okr: OKR) => {
  // 乐观更新（立即更新缓存）
  const okrs = getOKRs();
  const idx = okrs.findIndex(o => o.id === okr.id);
  if (idx >= 0) {
    okrs[idx] = okr;
  } else {
    okrs.push(okr);
  }
  okrsCache = okrs;
  localStorage.setItem(STORAGE_KEYS.OKRS, JSON.stringify(okrs));
  dispatchUpdate(); // 立即通知 UI 更新
  
  // 后台同步到服务器
  try {
    const existing = okrs.find(o => o.id === okr.id);
    if (existing) {
      await okrsAPI.update(okr.id, okr);
    } else {
      await okrsAPI.create(okr);
    }
    // 同步成功后，更新缓存时间戳
    cacheState.okrs.lastFetch = Date.now();
  } catch (error) {
    console.error('保存 OKR 到服务器失败:', error);
    // 可以选择回滚或显示错误提示
  }
};
```

**优点**：
- ✅ 保持原有同步 API 不变，前端代码无需修改
- ✅ 首次加载快速（从 localStorage 临时恢复，页面刷新场景）
- ✅ 后台自动同步，数据保持最新
- ✅ 页面刷新时快速恢复（使用 localStorage 临时缓存）
- ✅ 乐观更新，用户体验流畅
- ✅ 架构简洁，不依赖离线支持

**缺点**：
- ⚠️ 需要管理缓存状态
- ⚠️ 多标签页之间需要额外同步机制

#### 方案 B：响应式数据层（更现代）

**核心思想**：
- 使用 React Context + Hooks 管理数据状态
- 数据变更自动触发 UI 更新
- 统一的异步数据管理

**实现细节**：

```typescript
// hooks/useData.ts
import { useState, useEffect, useContext, createContext } from 'react';

// 数据上下文
const DataContext = createContext<{
  users: User[];
  okrs: OKR[];
  loading: { users: boolean; okrs: boolean };
  error: { users: Error | null; okrs: Error | null };
  refresh: () => Promise<void>;
}>(null!);

// 数据提供者
export const DataProvider: React.FC<{ children: React.ReactNode }> = ({ children }) => {
  const [users, setUsers] = useState<User[]>([]);
  const [okrs, setOkrs] = useState<OKR[]>([]);
  const [loading, setLoading] = useState({ users: false, okrs: false });
  const [error, setError] = useState({ users: null, okrs: null });

  const refresh = async () => {
    // 加载数据
    setLoading({ users: true, okrs: true });
    try {
      const [usersResult, okrsResult] = await Promise.all([
        usersAPI.getAll(),
        okrsAPI.getAll()
      ]);
      
      if (usersResult.success && usersResult.data?.users) {
        setUsers(usersResult.data.users);
      }
      if (okrsResult.success && okrsResult.data?.okrs) {
        setOkrs(okrsResult.data.okrs);
      }
    } catch (err) {
      setError({ users: err, okrs: err });
    } finally {
      setLoading({ users: false, okrs: false });
    }
  };

  useEffect(() => {
    refresh();
  }, []);

  return (
    <DataContext.Provider value={{ users, okrs, loading, error, refresh }}>
      {children}
    </DataContext.Provider>
  );
};

// 使用 Hook
export const useUsers = () => {
  const { users, loading, error } = useContext(DataContext);
  return { users, loading, error };
};

export const useOKRs = () => {
  const { okrs, loading, error } = useContext(DataContext);
  return { okrs, loading, error };
};
```

**优点**：
- ✅ 现代化的 React 模式
- ✅ 自动 UI 更新
- ✅ 统一的错误处理
- ✅ 更好的类型安全

**缺点**：
- ⚠️ 需要大量修改前端代码
- ⚠️ 不符合"最小改动"原则

#### 方案 C：混合方案（最佳实践）

**核心思想**：
- 结合方案 A 和方案 B 的优点
- 保持原有同步 API（向后兼容）
- 添加可选的响应式 Hook（新功能使用）

**实现策略**：
1. **保持原有同步函数**（`getUsers()`, `getOKRs()` 等）
2. **添加响应式 Hook**（`useUsers()`, `useOKRs()` 等）
3. **统一的数据管理层**（内存缓存 + 自动同步）
4. **渐进式迁移**（新组件使用 Hook，旧组件继续使用同步函数）

## 三、推荐方案：方案 A（智能缓存 + 自动同步）

### 3.1 为什么选择方案 A？

1. **最小改动**：前端代码几乎不需要修改
2. **向后兼容**：保持原有 API 不变
3. **性能优秀**：首次加载快速，后台自动同步
4. **用户体验**：乐观更新，响应迅速
5. **易于实现**：不需要大规模重构

### 3.2 实施步骤

1. **第一步**：实现内存缓存和后台同步逻辑
2. **第二步**：修改 `okrService.ts` 的同步函数，使用内存缓存
3. **第三步**：添加初始化函数，应用启动时调用
4. **第四步**：测试数据同步和错误处理
5. **第五步**：优化缓存策略和性能

### 3.3 关键优化点

1. **请求去重**：避免重复请求
2. **批量更新**：合并多个更新操作
3. **错误重试**：自动重试失败的请求
4. **页面刷新恢复**：使用 localStorage 临时缓存，快速恢复数据
5. **多标签页同步**：使用 BroadcastChannel API 或事件机制

## 四、总结

**推荐方案**：方案 A（智能缓存 + 自动同步）

**核心优势**：
- ✅ 最小化前端代码改动
- ✅ 保持原有 API 不变
- ✅ 性能优秀，用户体验好
- ✅ 易于实现和维护

**实施建议**：
1. 先实现核心功能（内存缓存 + 后台同步）
2. 逐步添加优化（请求去重、错误重试等）
3. 测试各种场景（离线、错误、多标签页等）
