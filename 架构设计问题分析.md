# OKR 系统后端架构设计 - 问题与挑战分析

## 🔴 严重问题（Critical Issues）

### 1. **状态机管理缺失 - 业务逻辑风险**

**问题描述**：
- OKR 有 10 种状态（DRAFT → PENDING_MANAGER → PENDING_GM → PUBLISHED → PENDING_ASSESSMENT_APPROVAL → PENDING_L2_APPROVAL → PENDING_L3_APPROVAL → PENDING_ARCHIVE → CLOSED）
- 当前设计只是简单的 `status` 字段，**没有状态转换规则验证**
- 前端可以任意修改状态，后端没有校验

**风险**：
- 用户可能跳过审批流程直接到 PUBLISHED
- 可能从 PENDING_L2_APPROVAL 直接跳到 CLOSED
- 状态回退没有规则限制
- 数据不一致导致业务逻辑错误

**建议**：
```typescript
// 需要定义状态转换规则
const STATUS_TRANSITIONS = {
  DRAFT: ['PENDING_MANAGER'],
  PENDING_MANAGER: ['DRAFT', 'PENDING_GM', 'PUBLISHED'],
  PENDING_GM: ['DRAFT', 'PUBLISHED'],
  PUBLISHED: ['PENDING_ASSESSMENT_APPROVAL'],
  // ... 完整的状态机
};
```

### 2. **审批权限验证缺失 - 安全漏洞**

**问题描述**：
- 前端通过 `getApproverRoles()` 计算审批人，但**后端没有验证**
- 任何用户都可以调用 `PUT /api/okrs/:id` 修改状态
- 没有验证当前用户是否有权限审批该 OKR

**风险**：
- 普通员工可以审批总经理的 OKR
- 跨部门审批没有限制
- 权限绕过导致数据被恶意修改

**建议**：
```typescript
// 后端必须验证审批权限
async function canApproveOKR(userId: string, okrId: string): Promise<boolean> {
  // 1. 获取 OKR 和用户信息
  // 2. 根据 workflow 计算审批人角色
  // 3. 验证当前用户角色是否匹配
  // 4. 验证部门匹配（如果需要）
  // 5. 验证状态是否允许审批
}
```

### 3. **数据冗余与一致性问题**

**问题描述**：
- `okrs.user_name` 冗余存储，用户改名后不一致
- `okrs.department` 冗余存储，部门调整后不一致
- `operation_logs.user_name` 冗余存储

**风险**：
- 历史数据不准确
- 审计追踪信息错误
- 报表数据失真

**建议**：
- 移除冗余字段，通过 JOIN 查询获取
- 或添加 `snapshot` 表记录历史快照
- 或使用触发器自动更新

### 4. **JSONB 查询性能问题**

**问题描述**：
- `objectives` JSONB 存储，查询和更新效率低
- 无法对 `keyResults` 建立索引
- 无法对 `selfScore`, `managerScore` 建立索引
- 复杂查询需要全表扫描

**风险**：
- 数据量大时查询慢
- 无法高效统计评分数据
- 报表生成性能差

**建议**：
```sql
-- 方案1: 规范化设计
CREATE TABLE objectives (
  id VARCHAR(50) PRIMARY KEY,
  okr_id VARCHAR(50) REFERENCES okrs(id),
  content TEXT,
  weight DECIMAL(5,2),
  self_score DECIMAL(5,2),
  manager_score DECIMAL(5,2),
  ...
);

CREATE TABLE key_results (
  id VARCHAR(50) PRIMARY KEY,
  objective_id VARCHAR(50) REFERENCES objectives(id),
  ...
);

-- 方案2: 如果坚持 JSONB，添加 GIN 索引
CREATE INDEX idx_okrs_objectives ON okrs USING GIN (objectives);
```

### 5. **并发更新冲突**

**问题描述**：
- 自动保存功能可能导致并发更新
- 没有乐观锁或悲观锁机制
- 多个用户同时修改同一 OKR 会覆盖

**风险**：
- 数据丢失
- 评分被覆盖
- 审批结果丢失

**建议**：
```sql
-- 添加版本号字段
ALTER TABLE okrs ADD COLUMN version INTEGER DEFAULT 1;

-- 更新时检查版本
UPDATE okrs 
SET ..., version = version + 1 
WHERE id = $1 AND version = $2;
```

## 🟡 重要问题（Important Issues）

### 6. **审批流程动态计算 - 性能与一致性**

**问题描述**：
- 审批人通过 `workflows` 表动态计算
- 每次查询都需要 JOIN 和计算
- 如果 workflow 修改，历史 OKR 的审批流程会变化

**风险**：
- 历史审批流程被错误修改
- 查询性能差
- 审批人变更导致已提交的 OKR 无法审批

**建议**：
```sql
-- 在 OKR 创建时快照审批流程
CREATE TABLE okrs (
  ...
  approver_l1_role VARCHAR(50),  -- 快照
  approver_l2_role VARCHAR(50),  -- 快照
  approver_l3_role VARCHAR(50),  -- 快照
  cc_roles TEXT[],                -- 快照
);
```

### 7. **缺少审计追踪**

**问题描述**：
- 只有 `operation_logs` 记录操作
- 没有记录**谁在什么时候修改了什么字段**
- 无法追踪状态变更历史
- 无法追踪评分变更历史

**风险**：
- 无法追溯问题
- 无法审计合规性
- 无法恢复误操作

**建议**：
```sql
-- 添加变更历史表
CREATE TABLE okr_history (
  id SERIAL PRIMARY KEY,
  okr_id VARCHAR(50),
  changed_by VARCHAR(50),
  changed_at TIMESTAMP,
  field_name VARCHAR(50),
  old_value TEXT,
  new_value TEXT,
  change_reason TEXT
);
```

### 8. **数据完整性约束不足**

**问题描述**：
- `objectives` 权重总和没有数据库约束
- `keyResults` 权重总和没有数据库约束
- 状态与数据一致性没有约束（如：PENDING_ASSESSMENT_APPROVAL 必须有 overallSelfAssessment）

**风险**：
- 数据不符合业务规则
- 前端验证可以绕过
- 数据不一致导致业务错误

**建议**：
```sql
-- 使用 CHECK 约束或触发器
ALTER TABLE okrs ADD CONSTRAINT check_objectives_weight 
CHECK (
  (SELECT SUM((obj->>'weight')::numeric) 
   FROM jsonb_array_elements(objectives) obj) = 100
);
```

### 9. **缺少事务管理**

**问题描述**：
- 状态变更可能涉及多个操作（更新 OKR、记录日志、通知等）
- 没有事务保证原子性
- 部分成功导致数据不一致

**风险**：
- 状态更新了但日志没记录
- 部分数据更新失败
- 数据不一致

**建议**：
```typescript
// 使用数据库事务
await db.transaction(async (trx) => {
  await updateOKRStatus(okrId, newStatus, trx);
  await addLog(action, module, details, trx);
  await notifyUsers(userIds, trx);
});
```

### 10. **API 设计不够 RESTful**

**问题描述**：
- `POST /api/grade-configs` 用于保存所有配置（应该是 PUT）
- `DELETE /api/workflows/:targetRole` 使用路径参数而非 ID
- 缺少专门的审批接口（应该用 `POST /api/okrs/:id/approve`）

**风险**：
- API 语义不清晰
- 不符合 REST 规范
- 难以扩展和维护

**建议**：
```typescript
// 审批接口
POST /api/okrs/:id/approve
POST /api/okrs/:id/reject
POST /api/okrs/:id/archive

// 状态变更接口
PATCH /api/okrs/:id/status
```

## 🟢 改进建议（Enhancement Suggestions）

### 11. **缺少数据验证层**

**问题描述**：
- 没有统一的数据验证
- 验证逻辑分散在 controller 中
- 没有使用验证库（如 Joi, Zod）

**建议**：
```typescript
// 使用 Zod 或 Joi 定义 Schema
const OKRSchema = z.object({
  title: z.string().min(1).max(500),
  objectives: z.array(...).min(1),
  // ...
});
```

### 12. **缺少缓存策略**

**问题描述**：
- 频繁查询 workflows, grade_configs, departments
- 没有缓存机制
- 每次请求都查数据库

**建议**：
- 使用 Redis 缓存配置数据
- 或使用内存缓存（Node-cache）
- 设置合理的过期时间

### 13. **缺少分页和过滤**

**问题描述**：
- `GET /api/okrs` 可能返回大量数据
- 没有分页机制
- 没有过滤和排序

**建议**：
```typescript
GET /api/okrs?page=1&limit=20&status=PUBLISHED&userId=u1&sort=created_at:desc
```

### 14. **错误处理不统一**

**问题描述**：
- 错误响应格式可能不一致
- 没有错误码体系
- 没有错误分类

**建议**：
```typescript
// 统一错误响应
{
  "success": false,
  "error": {
    "code": "OKR_NOT_FOUND",
    "message": "OKR not found",
    "details": {...}
  }
}
```

### 15. **缺少 API 版本控制**

**问题描述**：
- 没有 API 版本号
- 未来升级可能破坏兼容性

**建议**：
```
/api/v1/okrs
/api/v2/okrs
```
